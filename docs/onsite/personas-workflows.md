# ペルソナ別ワークフロー仮説

現場ヒアリング前の自己整理用。受託試験事業の典型的な 4 ペルソナを想定し、
各々の「一日の作業」「使うツール」「困りごと」「Matlens の介在余地」を仮置きする。

**この仮説は外して当然**。1〜2 回の現場対話で書き換えること。

---

## ペルソナ A: 試験案件 PM（プロジェクトマネージャ）

### 役割
- 顧客窓口、見積、進捗管理、納期コミット、品質責任
- 試験技師の作業をスケジューリングし、レポート提出までを束ねる

### 想定する一日

| 時間 | 行動 | 想定ツール |
|---|---|---|
| 09:00 | メール / Teams で顧客やりとり | Outlook |
| 09:30 | 進捗ホワイトボード前で立ち会い | 物理ボード |
| 10:00 | 既存 LIMS で当日の試験予定確認 | LIMS |
| 10:30 | 異常所見の検討会出席 | 会議室 |
| 13:00 | 新規見積（Excel テンプレ） | Excel |
| 15:00 | 納期遅延案件の優先度調整、技師との 1on1 | 口頭 |
| 16:30 | レポート最終レビュー、承認 | Word / PDF |

### 困りごと（仮説）
- 「全案件の進捗を一目で見たい」のに LIMS は案件単位の深い詳細しか出ない
- 納期リスクの早期警告がない（気づいたときには遅延）
- 試験技師の手があくタイミングが見えない
- 顧客から「今どうなってる？」と聞かれたとき即答できない

### Matlens の介在余地
- **Ops Dashboard** で全案件 KPI を 1 画面に
- **納期リスク 7 日以内** を毎朝確認
- **案件詳細** で specimen / test の進捗を顧客にスクショで送れる

### 確認質問
- 「LIMS のダッシュボード機能は使ってますか？」
- 「進捗管理のメインツールは何ですか？」
- 「顧客への状況報告はどんな頻度で？」

## ペルソナ B: 研究員（R&D / 材料エンジニア）

### 役割
- 顧客から相談された材料課題に対し、試験計画を立て、結果を解釈する
- 過去類似事例を引っ張って提案する
- 顧客との技術ディスカッションをリード

### 想定する一日

| 時間 | 行動 | 想定ツール |
|---|---|---|
| 09:00 | 文献調査（社内 OneDrive / 学術 DB） | PDF |
| 10:00 | 過去案件の結果検索（LIMS or 担当者の記憶） | LIMS |
| 11:00 | 顧客打合せ準備、想定質問整理 | PowerPoint |
| 14:00 | 新規試験計画立案、規格選定 | 紙＋ホワイトボード |
| 15:00 | 試験結果のクロス分析、考察文書作成 | Excel + Word |
| 17:00 | 翌日の優先順位リスト作成 | 紙メモ |

### 困りごと（仮説）
- 過去類似案件を見つけにくい（検索キーワードに依存）
- 試験種別 × 材料の経験有無を俯瞰したい
- 損傷事例を画像横断で検索したい
- 試験条件（温度・雰囲気）の組合せ別の結果分布を見たい

### Matlens の介在余地
- **試験マトリクス** で経験有無を一目で
- **損傷ギャラリー** で画像横断検索（タグフィルタ + 類似事例）
- **横断セマンティック検索** で「SUS316L 高温疲労 粒界破壊」のような自然言語
- **材料マスタ詳細** で過去試験履歴を一覧

### 確認質問
- 「過去事例の検索、最初に何を打ちますか？」（ライブで打ってもらう）
- 「論文・学会発表ベースの調査と、社内事例の調査は別ツール？」
- 「顧客が初めての材料を持ち込んだとき、どう調査スタートしますか？」

## ペルソナ C: 試験技師（オペレータ）

### 役割
- 受入試験片の準備（切り出し、研磨、寸法測定）
- 試験機操作、データ取得、一次評価
- 試験成績書の元データ整備

### 想定する一日

| 時間 | 行動 | 想定ツール |
|---|---|---|
| 08:30 | 受入試験片の検品、台帳記入 | 紙台帳 / LIMS |
| 09:00 | 試験機の校正・暖機 | 紙チェックリスト |
| 09:30 | 試験開始、データ取得（連続 3-5 件） | 試験機ソフト |
| 12:00 | 昼食、結果データの簡易確認 | Excel |
| 13:00 | 午後試験続行、異常があれば写真撮影 | 試験機 + スマホ |
| 16:00 | 試験片の戻し、保管棚に格納 | 物理棚 |
| 16:30 | 当日結果の入力（LIMS or Excel） | LIMS |
| 17:00 | 翌日試験の準備（治具・工具確認） | 物理 |

### 困りごと（仮説）
- 試験機ソフトと LIMS の二重入力（コピペ作業多い）
- 受入から保管までのトレーサビリティが紙頼み
- 異常があった試験の写真とデータの紐付けが面倒
- 同じ材料で過去どう試験したかすぐ出ない

### Matlens の介在余地
- **試験片トラッカー (Kanban)** で手元の状況を一画面
- **試験マトリクス** で「この材料の前回条件」を即参照
- **損傷ギャラリー** で異常所見を画像で記録、過去類似と並べる

### 確認質問
- 「試験機ソフトから LIMS への取り込みはどのくらい自動化されてますか？」
- 「受入試験片の物理管理（棚番号 / QR）は？」
- 「異常を発見したとき、誰にどう連絡しますか？」

## ペルソナ D: 切削プロセス研究者（Matlens の追加ターゲット）

### 役割
- 加工条件の最適化（速度・送り・工具選定）
- びびり / 工具寿命 / 表面性状の解析
- 新規工具の評価、CAM オペレータとのインターフェース

### 想定する一日

| 時間 | 行動 | 想定ツール |
|---|---|---|
| 09:00 | 前日の工具摩耗測定（顕微鏡） | 顕微鏡 + Excel |
| 10:00 | 切削条件の設計（過去データ + 経験） | Excel + 文献 |
| 11:00 | 試切り、加工力モニタの設置 | 試験機 + LabVIEW 等 |
| 13:00 | 試切り結果の解析（FFT、表面粗さ） | MATLAB / Python |
| 15:00 | CAM オペレータへの条件指示書作成 | Word / 紙 |
| 16:30 | 同僚との条件検討会 | 会議室 |

### 困りごと（仮説）
- 過去の切削条件と結果の組合せが Excel ファイル散在
- 工具寿命の予測根拠が経験値に偏る
- びびり発生条件の体系化ができてない
- 波形データ（切削抵抗・振動）の保管・横断比較ができてない

### Matlens の介在余地
- **切削条件エクスプローラ** で Vc × f 散布図 + びびり色分け
- **工具ライフトラッカー** で VB 進展 + Taylor 予測
- **波形ビューア** で時間領域 + FFT
- **Kc 見積 / Stability Lobe** パネルで経験式の可視化

### 確認質問
- 「impact test で modal params を測定する設備ありますか？」
- 「切削抵抗データの保管期間とフォーマットは？」
- 「Kc 表は社内独自値？工具メーカ提供？」
- 「CAM ソフトは何使ってますか？（NX / Fusion / hyperMILL 等）」

## ペルソナ間の役割分担まとめ

| ペルソナ | 主に使う画面 | 主に避ける画面 |
|---|---|---|
| PM | Ops Dashboard / 案件一覧 / レポート | 切削条件 / 工具ライフ |
| 研究員 | 試験マトリクス / 損傷ギャラリー / 横断検索 / 材料マスタ | 試験片トラッカー（粒度細かすぎ）|
| 試験技師 | 試験片トラッカー / 試験マトリクス / 損傷ギャラリー | Ops Dashboard（俯瞰ニーズ薄）|
| 切削研究者 | 切削条件エクスプローラ / 工具ライフ / 材料マスタ | 案件一覧（ビジネス側）|

→ **同じ画面でも見たい情報・操作粒度が違う**ので、デフォルトビュー / フィルタ初期値をペルソナで切り替えられると価値が上がる（Phase 4 候補）。

## ペルソナ仮説の検証方法

1. 初回ヒアリングで「貴社にこんな役割の人いますか？」と聞く
2. それぞれの代表者に 30 分ずつ「一日の流れを話してください」と振る
3. このファイルの想定と差分があれば該当行を**そのまま書き換え**
4. ペルソナ追加（例: QA 担当 / 設備管理 / 営業）が必要なら新規セクションを足す
