# ADR-015: 守秘義務対応とリポジトリ命名ポリシー

- ステータス: **Accepted**（2026-04-23 ユーザー方針として確定）
- 日付: 2026-04-23
- 関連 ADR: ADR-002（切削ドメイン分離）、ADR-007（連動更新ルール）、ADR-012（親密化戦略）

---

## 背景

Matlens は OSS として公開されており、**リポジトリに含まれる全ての内容が外部に閲覧可能**。一方で、開発者は特定クライアント案件に関与しており、守秘義務契約下にある。両立のためには、リポジトリに含める内容に明確な境界が必要。

これまで `docs/research/` 配下の一部ドキュメントに、仮名ベースの組織構造仮説と業界特定が容易な記述が共存していた。仮名は匿名化のつもりだったが、**業界特定性が高い記述と組み合わさる** と、想起可能性を十分に下げきれていなかった。2026-04-23 のユーザー方針で、より厳密な取り扱いに再定義する。

---

## 決定

### 1. リポジトリに入れないもの（絶対禁止）

- クライアント企業・団体の実名
- クライアント関連個人の氏名・役職・連絡先
- 契約形態（業務委託 / 準委任 / 派遣 等）の具体
- 契約金額・工数・期間の具体
- **仮名であっても**、業界・業種・拠点位置・親子会社構造等の推測を可能にする記述の組合せ
- クライアント組織の内部慣習（承認フロー・決裁者・人事情報）
- クライアント提供の資料・図面・測定データ
- ヒアリングで得た非公開情報
- スケジュール情報（開始日・マイルストーン・締切）

### 2. リポジトリに入れてよいもの

- 業界標準・公開規格の一般論（JIS / ASTM / ISO / AMS / MaiML 等）
- 公開情報ベースの技術知見（論文 / メーカ公式資料）
- 自作の技術実装・設計判断
- 匿名化された架空モックデータ
- **一般化された** ペルソナ仮説（「PM」「研究員」「試験技師」等、役割ベース）

### 3. ローカル保管（リポジトリ外）対象

以下はリポジトリに入れず `~/Documents/matlens-onsite-prep/` 等のローカル領域に保管する。

- 現場入り準備チェックリスト（個人の準備作業）
- 関係者マップ（実名・役職）
- ヒアリング用質問バンク（カテゴリ別）
- 自己紹介スクリプト（役割・経歴）
- ターゲットブリーフ（業界・組織仮説、非公開推察）

### 4. issue / PR / commit メッセージの命名ルール

- **OK**: 「現場」「連携先」「業界一般」「クライアント」「エンドユーザー」等の一般語
- **NG**: 特定企業名（実名・仮名問わず）、業界特定性が高い語の組合せ、プロジェクト固有コードネーム
- PR タイトル・issue タイトル・commit メッセージ全てに適用
- announcements.ts / README / 画面テキストも同様

### 5. 連携プロジェクトの扱い

peer として連携する他プロジェクト（例: `BoxPistols/machining-fundamentals`）は
**OSS として公開されている限り**、正式名で言及可能。ただし:

- 連携の背景や運用事情（誰が / なぜ / 何の目的で）で守秘義務に触れる情報は書かない
- 連携 ADR（ADR-012〜014）は設計判断ログとして具体名を含むが、これは OSS 間の公開連携として許容

### 6. 現場スケジュール情報の扱い

現場で共有されたスケジュール（開始日・各月の作業内容・マイルストーン）は **リポジトリに書かない**。ADR / issue / PR / announcements 全てに適用。

スケジュールを動機として作業優先順位を決めるのは問題ないが、**動機そのものは文書化しない**:

- OK: 「Phase 3.5 で主要画面を深化させる」
- NG: 具体的な月・日付・フェーズ名を含む動機記述、外部スケジュールを前提とした表現

動機を書きたい場合は、ローカル（`~/Documents/matlens-onsite-prep/`）に保管。

### 継続対応

- [ ] 新規 issue / PR / commit 作成時は必ず本 ADR を参照
- [ ] 既存 announcements.ts / README / PAGE_GUIDES に違反があれば都度修正
- [ ] CI で禁止語（特定企業名等）の検出を検討（Phase 4 で実装可能性）

### git 履歴残存に関する認識

本 ADR 施行前にリポジトリに含まれていた情報は、`git log` / `git blame` で依然として参照可能。GitHub 上でも過去のコミットツリーから旧ファイルを閲覧できる状態。これに対する本 ADR の立場は次のとおり:

- **削除前のドキュメントに実名は含まれていなかった** ため、新規の情報漏洩には至らない
- 仮名ベース + 業界推測可能記述の組合せは、採用・協業候補向けの OSS 閲覧レベルでは許容範囲
- 真に厳密な機密管理が必要になった場合は、`git filter-repo` による履歴書き換え + force-push を別 ADR で判断する
- 現時点では **履歴はそのまま残す**。新規コミットで本 ADR に従うことで、今後の差分は綺麗に保たれる

## 代替案と棄却理由

| 案 | 棄却理由 |
|---|---|
| **仮名のみ使用（頭文字による短縮形式を維持）** | 仮名 + 業界記述で想起可能性が残る。厳密な遮断が必要 |
| **リポジトリ全体を private 化** | OSS としての公開価値を失う（採用・協業候補への露出、技術実証）|
| **ビジネス仮説は書かない、技術のみ書く** | 方向性として正しいが、過去に書いたものを消す必要あり。本 ADR で明文化 |
| **全て匿名の "顧客 A" 等に置換** | 置換作業が大量、維持コスト高。文書自体をローカル移動する方が単純 |

## 影響

### ポジティブ
- OSS リポジトリを守秘義務下でも安全に運用可能
- 採用・協業候補が見たとき「この開発者は情報管理ができる」シグナル
- ビジネス情報はローカル、技術情報は OSS という役割分離が明確化

### ネガティブ
- ローカル保管物のバックアップ責任が個人にかかる（既に `~/Documents/matlens-onsite-prep/README.md` で対処方針あり）
- 技術選定の動機（なぜこの画面優先？）が repo 上で見えづらくなる
  → 動機は「業務領域の一般特性から」等の一般記述で置換可能
