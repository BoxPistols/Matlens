# ADR-002: 切削ドメインを別型・別画面として分離する

- ステータス: Accepted
- 日付: 2026-04-18
- 関連 issue / PR: #48, #66, #67

## 背景

Matlens は当初、受託試験（材料試験）事業を主対象として設計されていた。
しかし現場入り前のドメイン整理で、**切削プロセス研究** が事業上の重要軸として存在し、
「材料 × 試験」と「工具 × 切削条件 × 工作機械」は

- 物理量（応力・歪 vs 切削抵抗・送り・回転数）
- ライフサイクル（試験片受入〜廃棄 vs 工具再研磨・通算摩耗）
- 法規・規格（JIS/ASTM 試験規格 vs ISO 3685 工具寿命 / ISO 8688）
- 利用ペルソナ（試験技師・研究員 vs 切削プロセス研究者・CAM オペレータ）

が大きく異なることが判明した。

`Test` 型に工具・切削条件・びびり判定を詰め込むと、
材料試験側のフォーム UI がオプショナルフィールドで肥大化し、
切削側の一次解析（Taylor / Kc / SLD）も "試験の一種" として歪んだ表現になる。

## 決定

切削ドメインを **独立した型 / 独立した画面 / 独立した Repository** として分離する。

- 型: `src/domain/cutting/` に `CuttingCondition`, `ToolInstance`, `ToolWearSample`, `CuttingForceSample` を配置
- 画面: `src/features/cutting/` と `src/features/tools/` を新設し、案件・試験画面とは
  **サイドバーのセクションを分ける**（「受託試験」「切削プロセス」）
- Repository: `cuttingConditionsRepository`, `toolsRepository` を独立に DI 登録
- 計算モジュール: `src/features/cutting/utils/` に Taylor / Kienzle / Altintas SLD の純関数を配置し、
  他ドメインと共有しない

試験 (Test) と切削条件 (CuttingCondition) の紐付けは
**案件 (Project) 経由の弱結合** に留める（案件 ID を両者が持つ形）。

## 代替案と棄却理由

| 案 | 棄却理由 |
|---|---|
| `Test` 型にフィールド追加 | 型が 40+ フィールドに膨張。Form / Detail のどちらも切削専用フィールドを条件分岐で出し分ける必要があり、可読性が壊滅的 |
| `TestCategory = 'cutting'` で UI 分岐 | ライフサイクル（工具の再研磨・通算摩耗）が試験片モデルに合わない。一次解析 UI も根本的に異なる |
| 別アプリとして切り出す | 規格マスタ・材料マスタ・案件は共通。二重管理コストが過大 |

## 影響

### ポジティブ
- 受託試験側の型・UI はシンプルに保てる
- 切削側は Taylor / Kc / Stability Lobe という学術的に確立された枠組みに沿って拡張できる
- ペルソナ別のサイドバー構成が取れる（PM / 研究員 vs 切削研究者）

### ネガティブ
- 材料マスタ・規格マスタの双方向参照が発生（切削条件から材料へ、試験から材料へ）
- 「同じ材料の切削条件と試験結果を横串で見たい」という将来要件は
  Stage 2 で集計エンドポイント（ADR-010）として別途設計する必要がある
- サイドバー階層が一段深くなり、初見ユーザにはカテゴリ理解コストがかかる
  → Landing ページとオンボーディング導線で補う

## 将来の検討事項

- 「試験 × 切削」のクロスドメイン分析（例: 材料 X の引張試験結果と切削加工性の相関）が
  恒常的な業務要件になった場合、共通の「物性・加工性」ドメイン層を
  上位に設けることを検討する（ADR 改訂案件）
