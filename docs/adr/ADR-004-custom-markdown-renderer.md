# ADR-004: Markdown レンダラを自前実装する

- ステータス: Accepted
- 日付: 2026-04-18
- 関連 issue / PR: #45 (Reports 画面), PAGE_GUIDES 関連

## 背景

Matlens は次の場面で Markdown をレンダリングする。

1. **レポート画面** (`#/reports`): 試験報告書の本文
2. **PAGE_GUIDES**: 各画面のヘルプパネル
3. **お知らせ (announcements)**: リリースノート形式の一覧
4. **Storybook ドキュメント**: MDX 経由（こちらは対象外）

必要な要素は限定的で、

- 見出し（h1〜h3）
- パラグラフ
- 箇条書き（ul / ol、ネスト 1 段）
- 表（\| セパレート、行揃え記号対応）
- 強調（**bold**、_italic_）
- インラインコード（\`code\`）
- リンク（外部タブ開き）

であり、HTML 埋め込み・画像・カスタムディレクティブは不要。

## 決定

**純 TypeScript で最小限の Markdown パーサを自前実装する**（`src/utils/markdown.ts`）。

- 行指向パーサ（split by `\n` → ステート機械）
- インライン処理は正規表現のチェーン
- 出力は React 要素ツリー（生 HTML 文字列を挿入する API は使わない）
- XSS 対策は「HTML を解釈しない」ことで構造的に担保

## 代替案と棄却理由

| 案 | バンドル影響 (gzip) | 棄却理由 |
|---|---|---|
| `react-markdown` + `remark-gfm` | +40〜60 kB | 必要機能に対して過剰。remark プラグイン経由で XSS 対策・サニタイザが複雑化 |
| `marked` + `DOMPurify` | +25 kB | 生 HTML 挿入前提の経路を持ち込む必要があり、本プロジェクトの CSP 設計と相性悪 |
| `micromark` | +30 kB | 拡張性は高いが、必要機能には不釣り合い |
| MDX (@mdx-js) | 大 | そもそも本文は動的コンテンツでコンパイル不可 |

## 実装ポイント

- **入力はドメイン型から来る文字列** なので、書き手（Matlens 内部）を信頼できる。
  外部ユーザが自由に投稿する UGC ではない
- リンクは `href` を検証（`http(s):` / `mailto:` / 相対パスのみ許可）
- 表の列揃え（`:---`, `---:`, `:---:`）は CSS `text-align` に変換
- コードブロックはフェンス対応のみ（シンタックスハイライトは Story 内の `StoryCode` で別対応）
- パーサは純関数。入力 Markdown → 出力 React ノードで副作用なし
- テストは Vitest + React Testing Library。主要ノードのレンダリングを
  `getByRole('heading')`, `getByRole('table')` 等で網羅

## 影響

### ポジティブ
- バンドルサイズへの影響ゼロ（自前コードは木構造で数 KB）
- XSS 面積がゼロ（HTML 文字列を DOM に直接挿入する経路が存在しない）
- 機能追加が自由（プラグイン構造を気にしなくて良い）
- Matlens の他所で使うユーティリティ（テーブルセル内のインライン処理等）が再利用できる

### ネガティブ
- GFM 拡張（タスクリスト、取り消し線、脚注）を後から足すと実装コストがかかる
- 「完全な CommonMark 準拠」を期待されるとギャップになる
  → 書き手（Matlens 内部）に対し **サポート記法の一覧を docs 化** して境界を明示

## 将来の検討事項

- ユーザ入力 Markdown（例: 案件メモの任意記入欄）を解釈する要件が出た場合、
  サニタイザを別途挟む / あるいは `react-markdown` への切り替えを検討する
- コードブロックにシンタックスハイライトを付けるなら PrismJS の lazy load
  （Storybook 側で既に運用中のパターンを流用）
