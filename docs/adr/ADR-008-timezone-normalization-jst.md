# ADR-008: タイムゾーン正規化（内部 UTC / 表示 JST）

- ステータス: Accepted
- 日付: 2026-04-18
- 関連 issue / PR: #45, #48 ほか日時表示箇所全般

## 背景

Matlens は日本国内案件を主対象としつつ、将来的には海外 OEM との取引データも
扱う可能性がある。日時まわりで次のリスクが認識された。

- seeds の日時リテラルが `"2026-04-15T09:00:00"`（タイムゾーン指定なし）だと
  ブラウザのロケール依存で表示がずれる
- `Date.toLocaleString()` を直叩きすると
  サーバ / クライアントで結果が違うケースが発生（SSR は現状やっていないが、
  テスト環境と本番環境の差異も同根）
- 納期・受入日・試験実施日を跨ぐ集計で、境界日がブラウザ TZ 依存でブレる
- CSV 書き出し時にどのフォーマットで出すかが曖昧

## 決定

**内部は UTC（ISO 8601 `Z` サフィックス付き）で統一、表示は JST 固定** とする。

### ルール

1. **型レベルで ISO 文字列**
   - `createdAt`, `updatedAt`, `dueAt`, `startedAt`, `completedAt` は全て
     `"YYYY-MM-DDTHH:mm:ss.sssZ"` 形式（`Z` 必須）
2. **seeds は `Z` サフィックス付きで書く**
   - 例: `createdAt: "2026-01-15T00:00:00.000Z"`
3. **表示は `formatJst(iso, fmt)` ユーティリティ経由**
   - `src/utils/datetime.ts` に配置
   - 内部で `Asia/Tokyo` 固定（`Intl.DateTimeFormat` の `timeZone`）
4. **`new Date()` の直接利用を避け、`nowIso()` 経由** で ISO 文字列を取る
5. **期日境界は JST 00:00〜23:59 の範囲で判定**（UTC で切らない）
   - 納期リスク 7 日以内、今日の案件数などの集計
6. **CSV 書き出しは ISO 8601 のまま出す**（スプレッドシート側で解釈可能）

### 実装ユーティリティ

- `formatJst(iso, 'YYYY-MM-DD')` / `'HH:mm'` / `'full'` 等のフォーマット
- `toJstDateKey(iso)` — 集計キー用の `YYYY-MM-DD`（JST 基準）
- `isoWithin(isoStart, isoEnd, target)` — 境界判定
- `daysUntil(iso, refIso)` — 納期までの日数（JST 日付境界で計算）

## 代替案と棄却理由

| 案 | 棄却理由 |
|---|---|
| 全て JST で保存 | 海外 OEM データが混ざる将来要件で破綻。標準から外れる |
| ブラウザローカル時間 | テスト / CI / 別ロケールユーザで差分。境界日バグ量産 |
| Luxon / date-fns-tz 導入 | 表示要件が限定的（ほぼ `YYYY-MM-DD` / `HH:mm` のみ）。
  `Intl.DateTimeFormat` + 自前ラッパで依存ゼロを維持 |

## 影響

### ポジティブ
- seeds が TZ 揺れなく再現可能（ADR-003 決定論的 fixture と整合）
- 国際案件が来ても破綻しない
- 集計ロジックが決定論的（JST 基準のキー計算）

### ネガティブ
- `new Date()` 直叩きのレビュー観点が増える
  → ESLint ルール化を検討
- JST 固定なので海外拠点ユーザが見たときに違和感
  → 将来 `useTimezone()` フックで切替可能にする拡張余地を残す

## 将来の検討事項

- ユーザ設定のタイムゾーン（多拠点運用要件が出たとき）
- サマータイム圏の表示（EN/EU 拠点を想定する場合）
- MaiML エクスポートの日時表現（規格側の要求仕様を要確認）
