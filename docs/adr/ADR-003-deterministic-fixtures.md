# ADR-003: 決定論的 fixture の採用（seed 固定 + 手書き seeds 優先）

- ステータス: Accepted
- 日付: 2026-04-18
- 関連 issue / PR: #26, #45, #48

## 背景

Matlens は PoC 期間中、バックエンドが別チームに切り離されており、
UI 側は完全に mock データで動作する。モックデータの戦略として次の選択肢があった。

1. 完全ランダム (`faker.js` をビルド時ごとに新 seed で)
2. 決定論的ランダム（seed を固定した faker）
3. 手書き seeds（実データを模した 20〜50 件の静的 JSON）
4. 上記 2 と 3 の併用

初期の faker ランダム運用では、

- 同じ画面を 2 度開くと数値が変わり、スクリーンショットが安定しない
- Storybook のビジュアルリグレッションが毎回差分
- デモ時に「さっき見た案件」が消える / 順番が変わる → 顧客の体感が悪い
- テストで期待値を固定できず、「大体 N 件あれば OK」という緩い断言になる

という問題が顕在化した。

## 決定

**手書き seeds を主、固定 seed faker を従** とする。

- `src/mocks/seeds/*.ts` に **実ドメインを模した静的な seed データ**を配置
  - 材料 20 件、試験 40 件、案件 30 件、工具 15 件、切削条件 50 件など
  - ID 体系も決定論的（`MAT-0001`, `TEST-0001`, `IIC-2026-0001`）
- faker を使う場合は **必ず seed を固定**（`faker.seed(42)` を Generator の冒頭で）
- in-memory DB に積む初期データは seeds を基に、
  量的拡張が必要なものだけ Generator で決定論的に増幅
- **時刻フィールドは固定基準日**（`createdAt` は seed 側で `2026-01-15T...` 等の
  リテラル、相対時刻は基準日からオフセット）

## 代替案と棄却理由

| 案 | 棄却理由 |
|---|---|
| 完全ランダム | ビジュアルリグレッション、デモ再現性、テスト信頼性が壊れる |
| 固定 seed faker のみ | 量は出るが「このドメインらしさ」が出ない（材料名がランダム英数字等） |
| 実データの匿名化コピー | PoC 段階では顧客実データが手に入らない。現場入り後も機密管理が必要で
  リポジトリに入れる判断は別レベル |

## 実装ポイント

- seeds はドメイン側の型 (`Material`, `Test`, `CuttingCondition`) を直接返す
  （DTO 変換なし）
- in-memory DB は `structuredClone` で seeds を複製し、mutation は
  クローン側のみに適用（テスト間の汚染防止）
- MSW handler は in-memory DB を参照し、REST 実装時に同じレスポンス形状を再現
- 現場ヒアリング後、実データのパターンが判明した時点で seeds を書き換える
  **前提で** 命名は実値に近いが、数値レンジは意図的に平均化

## 影響

### ポジティブ
- デモ・Storybook・E2E が全て再現可能
- テストで「案件 ID `IIC-2026-0001` は状態 `testing` である」のような強い断言が書ける
- ヒアリング結果を seeds に即反映する文化が生まれた（`docs/onsite/` と双方向）

### ネガティブ
- seeds が更新されると依存テストの期待値も追従修正が必要
  → 統計的性質に対するテストは別途 Generator + 固定 seed で担保
- 量的ストレステスト（1 万件 UI）は別 Generator を用意する必要あり
  （現状 `performance.stories.tsx` で限定的に実施）

## 将来の検討事項

- 現場でヒアリングした実データ断片を seeds に反映する際、
  匿名化ポリシー（顧客名・案件番号の扱い）を別 ADR として整理する
