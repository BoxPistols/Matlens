# ADR-007: PAGE_GUIDES / announcements / README の連動更新ルール

- ステータス: Accepted
- 日付: 2026-04-18
- 関連 issue / PR: #45, #48, #50 ほか複数 PR で運用中

## 背景

Matlens は機能追加が短サイクルで、画面が増減する。しかし過去の PR では
次の乖離が頻発した。

- 新画面を追加したのに **お知らせページに掲載されていない**
  → リリースノートとしての連続性が欠落
- 新画面を追加したのに **PAGE_GUIDES に説明がない**
  → ユーザが画面を開いても何をする画面か分からない
- README の「できること」セクションと **実機能が乖離**
  → リポジトリ読者（採用・協業候補）の理解が遅れる

PoC 段階では読み手（本人・レビュアー・将来の運用者）の数が少ないからこそ、
情報源のズレが「信頼性」の重大毀損になる。

## 決定

**新画面 / 主要機能を追加・変更する PR では、次の 3 ファイルを必ず同一 PR 内で更新する**。

1. `src/data/announcements.ts` — お知らせ / リリースノート
2. `src/data/pageGuides.ts`（または該当 PAGE_GUIDES 定義） — 画面内ヘルプパネル
3. `README.md` — 「画面一覧」「主要機能」セクション

運用ルール:

- **「機能が動いているがドキュメントが未更新」の状態で PR を merge しない**
- レビュー観点に「連動更新が揃っているか」を明示
- 画面削除時は 3 箇所から **まとめて除去**（残滓を残さない）

## 代替案と棄却理由

| 案 | 棄却理由 |
|---|---|
| 別 PR でドキュメント更新 | 並列運用で必ず忘れる。PoC 規模で 3 ファイル追加程度は負担にならない |
| README を自動生成 | 生成スクリプトの信頼性担保コストが、手で書くコストを上回る |
| Changelog ツール（changesets / release-please） | 日本語運用・多言語化の選択肢を狭める。お知らせは
  ユーザ向けコピーで、機械生成の形式と馴染まない |

## 実装ポイント

- `announcements.ts` はドメインデータと同じ扱い（型あり・seed 形式）
  - `id`, `publishedAt`, `title`, `body` (Markdown), `tags`, `screens`
- `pageGuides.ts` は画面 path → ガイド本文の map
  - サイドバーから該当画面を開くとヘルプパネルに表示される
- README は日本語セクションのみ更新（英語版は現時点で未整備）
- 連動更新は **機械的チェックではなく PR テンプレートのチェックリスト** で担保

## 影響

### ポジティブ
- リリースノートとしての連続性が保たれる
- 画面追加・削除の歴史がお知らせページから追える
- README が現状と常に整合 → 初見読者の信頼が得られる

### ネガティブ
- PR あたりの作業量が増える（3 ファイル × 日本語文面）
- 文面の質が PR ごとに揺れる（→ 将来はテンプレ化を検討）
- 手動運用なので、急ぎの hotfix PR で抜けるリスク
  → **抜けた場合はフォロー PR で即復元** を運用ルール化

## 将来の検討事項

- 英語版お知らせ / README の整備（バイリンガル化の全体方針と連動）
- お知らせと PR のリンクを双方向に（PR 本文に announcement id、announcement に PR URL）
- 画面追加時のスキャフォールド（テンプレファイル生成）スクリプト
